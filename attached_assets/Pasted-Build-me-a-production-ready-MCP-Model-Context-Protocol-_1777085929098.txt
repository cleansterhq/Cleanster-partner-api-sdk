Build me a production-ready MCP (Model Context Protocol) server for Cleanster, a B2B cleaning services marketplace. The server should let AI assistants like Claude interact with the Cleanster platform programmatically.

## Stack
- TypeScript
- Official Anthropic MCP SDK (@modelcontextprotocol/sdk)
- Express for the HTTP transport layer
- Zod for input validation
- Axios for calling the existing Cleanster REST API

## Architecture
The MCP server is a thin translation layer between MCP tool calls and our existing Cleanster REST API (documented at postman.com/cleanster-api). It does NOT talk to the database directly. It owns:
- MCP protocol handling
- Auth scoping (per-user, per-tenant)
- Input validation
- Response formatting

## Auth
For v1, use bearer token (API key) auth via an Authorization header. Design it so OAuth 2.0 with PKCE can be swapped in later without rewriting tool handlers — pass the token through to the underlying Cleanster API on every call. Include a TODO marking where the OAuth flow will plug in.

## Tools to expose (v1)
Read-only first:
1. list_bookings — args: property_id (optional), status (optional), date_from (optional), date_to (optional), limit (default 20)
2. get_booking — args: booking_id
3. list_properties — args: account_id (optional), property_type (optional: apartment, house, str, commercial)
4. get_property — args: property_id
5. list_cleaners — args: region (optional), available_on (optional date)
6. get_payout_records — args: cleaner_id (optional), date_from, date_to

Write operations:
7. create_booking — args: property_id, service_type (enum: apartment, house, str, commercial, chores, handyman), scheduled_at (ISO datetime), notes (optional), checklist_id (optional)
8. cancel_booking — args: booking_id, reason (optional)
9. reschedule_booking — args: booking_id, new_scheduled_at
10. assign_crew — args: booking_id, cleaner_ids (array)
11. update_checklist — args: booking_id, cleaner_id, checklist_items (array of {task, required})

## Project structure
```
/src
  /tools          — one file per tool, each exports name, description, schema, handler
  /api            — Cleanster REST API client (axios wrapper with auth)
  /auth           — token validation, scope enforcement
  server.ts       — MCP server bootstrap, tool registration
  index.ts        — entry point
/tests            — vitest unit tests for each tool handler with mocked API
.env.example
README.md
```

## Requirements
- Each tool file should be self-contained: schema (Zod), handler, and a clear description string the AI will see
- The Cleanster API client should be a single class with methods per endpoint, taking a token argument
- Mock the Cleanster API in tests — do not hit real endpoints
- Include rate limiting (express-rate-limit) — 60 req/min per token
- Include request logging (pino) with token redaction
- Include a /health endpoint
- README should cover: local setup, env vars, how to connect from Claude Desktop (claude_desktop_config.json snippet), how to add a new tool
- .env.example with: CLEANSTER_API_BASE_URL, MCP_SERVER_PORT, LOG_LEVEL

## Constraints
- Do not invent Cleanster API endpoints — leave the REST URLs as configurable constants in /api/endpoints.ts with TODO comments noting they need to be confirmed against the real API
- Do not implement OAuth in v1, but leave a clean seam for it
- Keep tool descriptions concise and action-oriented — they are what the AI reads to decide when to call the tool
- Return structured JSON in tool responses, not prose
